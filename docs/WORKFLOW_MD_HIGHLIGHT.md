# MD 高亮渲染工作流程

> 替代方案：绕开 PDF 坐标问题，直接在 Markdown 层面进行高亮渲染

## 背景与动机

旧方案（PDF坐标驱动）遇到一个关键瓶颈：**部分PDF扫描完成后，文字以图片形式存在，没有可选择文本层**。这导致 PyMuPDF 无法提取坐标，高亮功能失效。

新方案改为**文本驱动**：直接对 MinerU 输出的 `full.md` 进行高亮处理，然后渲染成 PDF，彻底解决图片型 PDF 的支持问题。

---

## 架构对比

```
【旧方案 - 坐标驱动】                    【新方案 - 文本驱动】
                                             
PDF ──→ MinerU ──┬──→ layout.json ──┐        PDF ──→ MinerU ───→ full.md ───┐
                 │                    ↓                                      ↓
                 └──→ full.md ──→ LangExtract ──→ 实体+坐标 ──→ PyMuPDF ──→ PDF高亮
                                                           
   ⚠️ 问题: 图片型PDF无文本层                    ✅ 优势: 无视PDF类型
            坐标匹配复杂                               直接文本替换
            layout.json粒度问题                        支持跨行高亮
```

## 关键改进点

| 方面 | 旧方案 | 新方案 |
|------|--------|--------|
| **PDF类型支持** | ❌ 只支持文本型PDF | ✅ 支持所有类型 |
| **坐标依赖** | 依赖 layout.json | 无需坐标 |
| **高亮精度** | span级别，可能偏移 | 文本级别，精确匹配 |
| **跨行高亮** | 困难 | 自然支持 |
| **代码块保护** | 需额外处理 | 自动跳过 |
| **实现复杂度** | 高(匹配算法复杂) | 低(文本替换即可) |
| **目录结构适应性** | 固定路径 | 智能发现 |

## 使用示例

```python
from md_highlight_pipeline import MDHighlightPipeline

# 初始化 Pipeline
pipeline = MDHighlightPipeline(
    mineru_output_dir="mineru_output"
)

# 处理指定任务 (自动查找文件)
result = pipeline.process(
    task_id="513f81dc-4fca-42b3-a4a9-0d58d99db2d2",
    output_path="highlight_output/report.pdf"
)

print(f"✅ 高亮PDF已生成: {result.output_path}")
print(f"📊 提取实体数: {result.extraction_count}")
print(f"🎨 高亮次数: {result.highlight_count}")
```

## 注意事项

1. **字体配置**: CSS 中的 `font-family` 必须匹配系统已安装的中文字体，否则中文会显示为方框
2. **GTK3 (Windows)**: 如果生成 PDF 失败，请检查是否安装了 GTK3 运行时
3. **关键词冲突**: 长词优先匹配策略确保 "人工智能技术" 优先于 "人工" 被匹配
4. **代码块保护**: 自动跳过 `pre`, `code` 标签内的文本，避免代码被误高亮
