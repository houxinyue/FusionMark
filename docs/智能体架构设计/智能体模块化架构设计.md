# 智能体模块化架构设计

## 1. 目标

本方案面向 FusionMark 的“配置智能小助手”能力，目标是把智能体做成一个**独立应用**，与主解析服务解耦，能够单独启动、单独开发、单独部署。

系统要解决的核心问题是：

- 支持用户通过多轮对话生成和修改配置草稿
- 支持会话状态恢复和历史回放
- 支持短期运行状态和长期归档分离
- 支持后续接入 LangGraph，但不强依赖
- 支持提示词独立管理和版本化

---

## 2. 设计原则

1. **独立应用原则**  
   智能体必须放在单独目录中，作为一个独立应用启动，不揉进主服务目录。

2. **职责隔离原则**  
   交互、编排、存储、提示词、校验各自独立，不互相穿透。

3. **运行时与归档分离原则**  
   Redis 只存运行时状态，MinIO 只存长期归档。

4. **提示词资产化原则**  
   系统提示词、模板、示例统一放在 `prompts/` 下，不散落在代码中。

5. **可演进原则**  
   第一版先用普通服务编排，后续可平滑迁移到 LangGraph。

---

## 3. 总体架构

```text
用户
  │
  ▼
交互模块
  │  接收输入 / 展示消息 / 展示草稿 / 触发确认
  ▼
多轮对话模块
  │  意图识别 / 上下文召回 / 草稿生成 / 校验 / 状态推进
  ▼
存储模块
  ├─ Redis：会话状态、checkpoint、恢复点
  └─ MinIO：会话归档、审计记录、回放文件
```

### 模块边界

| 模块 | 职责 | 不负责 |
|---|---|---|
| 交互模块 | 接收消息、展示对话、展示草稿、发起确认 | 不直接访问 Redis / MinIO |
| 多轮对话模块 | 推进会话、生成草稿、校验、追问、挂起确认 | 不负责界面渲染 |
| 存储模块 | 保存状态、恢复状态、归档会话 | 不参与业务推理 |
| 提示词模块 | 管理系统提示词、模板、示例 | 不写业务逻辑 |

## 5. 存储模块设计

存储模块只负责“记忆”和“恢复”，不负责业务判断。

### 5.1 Redis 设计

Redis 用于短期运行状态。

#### Key 设计

```text
agent:{userId}:{sessionId}:checkpoint_list
agent:{userId}:{sessionId}:checkpoint:{checkpointId}
agent:{userId}:{sessionId}:session_meta
```

#### 类型设计

| Key | 类型 | 用途 |
|---|---|---|
| `...:checkpoint_list` | ZSET | checkpoint 顺序、最新状态定位 |
| `...:checkpoint:{id}` | HASH | 单次状态快照 |
| `...:session_meta` | HASH | 会话元信息 |

#### 建议字段

`session_meta`：

```text
session_id
user_id
title
status
current_step
created_at
updated_at
```

`checkpoint:{id}`：

```text
checkpoint_id
parent_checkpoint_id
node_name
current_step
state_version
payload
created_at
updated_at
```

`payload` 保存序列化后的状态内容，例如：

- 对话消息
- 当前草稿
- 校验结果
- 待确认动作
- 最近一次错误信息

#### TTL

建议 10 天左右，保证：

- 支持短期恢复
- 不让 Redis 无限膨胀
- 长期归档交给 MinIO

---

### 5.2 MinIO 设计

MinIO 用于长期归档和回放。

#### 路径结构

```text
{project}/{env}/agent/{userId}/session/{sessionId}.jsonl
```

#### 归档原则

不要直接把 Redis 原始结构照搬到 MinIO。  
MinIO 应存稳定的业务归档格式，建议包括：

- 会话元信息
- 消息历史
- checkpoint 摘要
- 最终草稿
- 审计信息
- 决策摘要

#### 示例结构

```json
{
  "schema_version": "1.0",
  "project": "fusion-mark",
  "env": "dev",
  "user_id": "u1001",
  "session_id": "s1001",
  "created_at": "2026-05-27T10:00:00Z",
  "updated_at": "2026-05-27T10:05:00Z",
  "messages": [],
  "checkpoints": [],
  "final_draft": {},
  "summary": {}
}
```

## 6. 交互模块设计

交互模块负责用户体验，不直接依赖存储实现。

### 6.1 职责

- 接收用户输入
- 展示历史会话
- 展示当前草稿
- 展示校验结果
- 触发“保存 / 激活 / 继续修改”
- 恢复历史会话

### 6.2 页面布局建议

如果集成到前端页面，可以采用三栏布局：

- 左侧：会话列表
- 中间：对话区
- 右侧：草稿与校验区

### 6.3 交互原则

- 草稿始终可见
- 关键动作必须确认
- 结果可回放
- 会话状态要可解释

---

## 7. 多轮对话模块设计

多轮对话模块是智能体核心。

### 7.1 职责

- 意图识别
- 历史上下文召回
- 草稿生成
- 草稿修正
- 校验草稿
- 人工确认前挂起
- 状态推进

### 7.2 状态模型

```python
class AgentSessionState:
    session_id: str
    user_id: str
    messages: list
    current_step: str
    current_draft: dict
    retrieved_context: list
    pending_action: str | None
    checkpoint_id: str | None
    last_error: str | None
```

### 7.3 状态流转

```text
开始
  ↓
意图识别
  ↓
上下文召回
  ↓
草稿生成
  ↓
草稿校验
  ├─ 通过 → 等待确认
  └─ 失败 → 追问 / 修正
  ↓
用户确认
  ↓
保存 / 激活 / 归档
```

### 7.4 LangGraph 预留点

后续如果接入 LangGraph，可以自然映射为：

- `intent_gatekeeper`
- `context_retriever`
- `draft_generator`
- `draft_validator`
- `human_review_interrupt`
- `commit_executor`

第一版不强制使用 LangGraph，但状态边界要按照图式思维预留。

## 8. 提示词模块设计

提示词必须作为独立资源管理。

### 8.1 目录划分

```text
app/prompts/
├── system/
├── templates/
└── examples/
```

### 8.2 目录用途

| 目录 | 用途 |
|---|---|
| `system/` | 系统提示词，定义角色、边界、输出格式 |
| `templates/` | 可复用任务模板，支持参数填充 |
| `examples/` | few-shot 示例和参考样例 |

### 8.3 管理原则

- 提示词使用 Markdown 文件存储
- 通过统一的 PromptProvider 加载
- 不把长提示词直接写死在业务代码中
- 便于版本管理和审查

## 9. 接口设计

建议接口围绕“会话”来组织。

```text
POST   /api/v1/agent/sessions
GET    /api/v1/agent/sessions/{sessionId}
POST   /api/v1/agent/sessions/{sessionId}/messages
POST   /api/v1/agent/sessions/{sessionId}/confirm
POST   /api/v1/agent/sessions/{sessionId}/restore
GET    /api/v1/agent/sessions/{sessionId}/checkpoints
```

### 接口职责

| 接口 | 职责 |
|---|---|
| `sessions` | 创建/读取会话 |
| `messages` | 推进对话状态 |
| `confirm` | 执行用户确认动作 |
| `restore` | 恢复会话状态 |
| `checkpoints` | 查看历史快照 |

## 10. 代码模块建议

### 10.1 `app/main.py`

应用入口，负责启动服务、注册路由、加载配置。

### 10.2 `app/core/service.py`

对外服务门面，封装一次完整对话处理流程。

### 10.3 `app/core/orchestrator.py`

编排层，负责调用意图识别、检索、生成、校验和存储。

### 10.4 `app/agent/*`

核心智能逻辑：

- `intent_gatekeeper.py`
- `context_retriever.py`
- `draft_generator.py`
- `draft_validator.py`
- `human_review.py`

### 10.5 `app/storage/*`

存储封装：

- `session_store.py`
- `checkpoint_store.py`
- `archive_store.py`

### 10.6 `app/config/settings.py`

统一读取环境变量，例如：

- Redis 地址
- MinIO 地址
- 项目名
- 环境名
- 对话模型配置

## 12. 实施顺序

建议按下面顺序开发：

```text
1. 搭独立应用目录
2. 定义会话模型和状态结构
3. 实现 Redis session / checkpoint 存储
4. 实现 MinIO 归档
5. 实现提示词目录和加载器
6. 实现多轮对话编排
7. 实现 HTTP API
8. 接前端页面
9. 补测试
```

---

## 13. 验收标准

第一版完成时，应满足：

1. 智能体可以独立启动
2. 能创建会话并发送消息
3. 能生成配置草稿
4. 能保存 checkpoint
5. 能从 checkpoint 恢复
6. 能把会话归档到 MinIO
7. 提示词文件独立管理
8. 主服务不需要直接依赖智能体内部实现

---

## 14. 结论

这套方案的核心是把智能小助手做成一个**独立、可启动、可恢复、可归档**的应用。

最终形态应该是：

- 独立目录
- 独立启动
- 独立存储
- 独立提示词
- 独立对话状态

这样后续无论是接 LangGraph，还是替换对话模型、存储后端、前端交互方式，都不会把主服务结构拖乱。
