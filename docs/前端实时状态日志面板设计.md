# 前端实时状态日志面板设计

## 背景

当前前端左侧区域主要以“进度条 + 三段步骤点”的形式展示任务处理状态。但后端已经切换为基于 Redis 的分阶段进度结构，返回的数据不再只是简单百分比，而是包含：

- `stage`
- `stage_progress`
- `overall_progress`
- `mineru`
- `extraction`
- `highlight`

其中每个阶段还包含：

- `state`
- `progress`
- `logs`

这意味着后端已经具备表达“当前执行到哪一步、这一阶段发生了什么”的能力，而前端仍然试图把复杂状态压缩成一个单一进度条，导致表现失真、维护困难，也不利于定位任务卡在哪个阶段。

## 目标

将左侧“进度条卡片”改造成“实时状态日志面板”，重点展示：

1. 当前任务的整体状态
2. 当前所处阶段
3. 每个阶段的运行状态
4. 来自后端的实时日志

## 非目标

本次改造不处理以下内容：

1. 不修改后端 WebSocket 消息协议
2. 不把完整日志持久化到 Redis
3. 不引入复杂图表或时间轴组件
4. 不删除 `legacy` 兼容逻辑，只弱化其主路径地位

## 当前问题

### 1. 展示模型与后端数据模型不匹配

前端旧逻辑假设状态是线性进展的，依赖推导函数根据少量字段估算进度百分比。但 Redis 架构下，任务是典型的“阶段式处理”：

- MinerU 解析
- 实体提取
- 高亮渲染

每个阶段都有自己的状态和日志，强行压缩成单进度条会造成：

- 用户难以理解当前究竟卡在哪一步
- 百分比并不真实
- 前端代码中出现新旧两套逻辑并存

### 2. 当前 UI 对“长等待阶段”不友好

实体提取阶段可能需要等待大模型调用 1-3 分钟，这种情况下单纯显示一个百分比意义有限。更有价值的是直接告知用户：

- 当前处于“实体提取”
- 正在调用模型
- 这是预期等待，不是卡死

### 3. 缺乏面向排查的问题信息

当任务失败或异常耗时时，用户和开发者都需要看到最近发生过哪些状态更新。当前纯进度条形式无法满足这个需求。

## 方案概述

保留一个轻量的总体进度条作为辅助视觉，不再把它作为核心信息承载。左侧卡片调整为以下结构：

1. 状态摘要区
2. 阶段状态区
3. 实时日志区

## 界面结构

### 状态摘要区

展示：

- 当前总体状态：`pending / processing / completed / failed`
- 当前阶段：`准备中 / MinerU 解析 / 实体提取 / 高亮渲染 / 完成 / 失败`
- 总体进度：`overall_progress`

建议形式：

- 标题：`处理状态`
- 副信息：`当前阶段：实体提取`
- 轻量进度条：仅展示整体进度

### 阶段状态区

展示三行阶段状态：

1. MinerU 解析
2. 实体提取
3. 高亮渲染

每行包含：

- 阶段名称
- 当前状态
- 当前阶段百分比

状态枚举：

- `pending`
- `running`
- `completed`
- `failed`

### 实时日志区

作为本次改造的核心展示区。

日志内容来自：

- `progress.mineru.logs`
- `progress.extraction.logs`
- `progress.highlight.logs`

展示形式：

- 滚动容器
- 每条日志带阶段前缀
- 失败日志使用强调颜色
- 新日志自动滚动到底部

示例：

```text
[MinerU] 准备解析文档...
[MinerU] 解析中... 第 3/12 页
[Extraction] 正在调用大模型...
[Extraction] 分析文档内容（此步骤耗时较长，请耐心等待）...
[Highlight] 开始渲染...
```

## 前端状态模型

在 `frontend/src/app.js` 的全局 `state` 中新增日志相关状态：

```js
{
  logs: [],
  logKeys: new Set()
}
```

其中：

- `logs` 用于当前任务的日志列表展示
- `logKeys` 用于前端去重，避免同一条阶段快照日志被重复追加

建议日志项结构：

```js
{
  key: "mineru|准备解析文档...",
  stage: "mineru",
  text: "准备解析文档...",
  level: "info",
  time: "14:32:18"
}
```

## 日志追加策略

由于当前后端发送的是“阶段快照”而不是“日志事件流”，前端需要自己做“抽取 + 去重 + 追加”。

### 规则

1. 新任务开始时，清空 `state.logs` 和 `state.logKeys`
2. 每次收到 WebSocket `progress` 消息：
   - 读取三个阶段的 `logs`
   - 转为统一日志项
   - 使用 `stage + text` 作为去重键
   - 未出现过的日志才追加到 `state.logs`
3. 任务完成或失败后，保留日志，不自动清空

### 说明

当前使用前端内存存放实时日志，是一个最小可用方案。这样无需改后端协议，也足够满足本次“实时展示当前解析状态”的目标。

## 与现有逻辑的关系

### 保留

保留：

- WebSocket 实时监听机制
- 后端现有 Redis 进度结构
- 顶部轻量进度条

### 弱化或停用

弱化以下旧逻辑的主路径角色：

- `calculateProgressPercent()`
- `updateStepsByProgress()`

这些逻辑主要服务于旧式推导进度模型，后续只作为兼容兜底使用，不再作为新结构的主处理逻辑。

## 用户体验预期

改造后，用户在处理过程中能明确看到：

1. 当前任务卡在哪个阶段
2. 当前阶段正在做什么
3. 是否只是模型调用耗时，而不是页面卡住
4. 失败时最后一批日志是什么

相比旧进度条方案，新的日志面板更适合此类阶段式任务，也更接近真实后端处理状态。

## 后续扩展：日志落 Redis

本次先将日志追加存放在前端内存。后续为了支持：

- 页面刷新后恢复日志
- 历史任务查看日志
- 多客户端查看相同任务日志

可以考虑把完整日志存入 Redis。

### 推荐方向

在现有任务状态结构之外，为每个任务增加日志列表存储，例如：

- `fusionmark:task:logs:{task_id}`

每条日志作为追加记录写入 Redis List 或 Stream。

### 建议格式

```json
{
  "stage": "extraction",
  "level": "info",
  "message": "正在调用大模型...",
  "timestamp": "2026-04-24T14:32:18+08:00"
}
```

### 好处

1. 页面刷新后可恢复日志
2. 不依赖前端内存
3. 支持后端主动查询和排障
4. 为未来“任务历史详情页”提供基础能力

## 实施顺序建议

1. 重构左侧卡片 DOM 结构
2. 引入前端日志状态与去重追加逻辑
3. 基于新进度结构更新摘要区和阶段状态区
4. 接入日志列表渲染和自动滚动
5. 保留旧逻辑作为兜底，不作为主路径
6. 完成样式调整与流程验证

## 验收标准

1. 提交任务后，左侧卡片能显示当前阶段和整体状态
2. WebSocket 更新时，日志能实时追加
3. 同一条日志不会重复刷屏
4. 完成或失败后，日志仍可保留查看
5. 页面不再依赖旧式推导进度条才能理解任务状态
6. 现有后端接口与 WebSocket 协议无需修改即可工作
