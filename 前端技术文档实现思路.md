# Project Context

## Purpose

FusionMark Frontend 是一个基于 Web 的 PDF 智能解析与高亮系统的前端应用程序。它配合 FusionMark 后端服务，实现了多模态文档的可视化交互。用户可以直接在 Web 浏览器中上传文档、实时追踪 AI 解析与实体提取进度，并精准查看带有彩色高亮标注的原生排版 PDF 结果。

### 核心特性

* **精准高亮渲染**：基于底层 PDF 引擎，动态计算文本与坐标映射，在保留 PDF 原始排版的基础上，实现提取实体的精准彩色高亮覆盖。
* **实时进度反馈**：深度集成 WebSocket 通信，打通与后端的实时链路，将后端的异步任务进度无缝推送到前端 UI 进行动态展示。
* **配置驱动体验**：提供可视化的配置管理面板，支持灵活调整提示词模板（Prompt）和高亮类别颜色（如克莱因蓝、爱马仕橙等专属配色）映射。

## Tech Stack

### 核心框架

* **Nuxt 3** - 现代化 Vue 全栈/同构框架，提供企业级目录规范和自动导入
* **Vue 3.x** - 渐进式 JavaScript 框架（Composition API 优先）
* **TypeScript ~5.3.0** - 静态类型检查，保障复杂的 PDF API 调用和数据接口安全
* **Pinia ^2.1.0** - Vue 状态管理库，管理全局任务、配置和 WebSocket 状态

### UI 组件库与样式

* **Element Plus ^2.x** - 核心业务组件库（表单、进度条、上传组件等）
* **Tailwind CSS ^3.4** - 实用优先的 CSS 框架，用于快速构建深色科技风和自定义主题色配置

### PDF 解析与高亮相关

* **pdfjs-dist ^4.x (Mozilla)** - 核心 PDF 渲染引擎，负责 Canvas 绘制与底层文本层（TextLayer）解析
* **pdf-text-search (自定义/生态库)** - 用于在 PDF 文本层中定位提取实体的精确 DOM 坐标

### 工具库

* **@vueuse/core ^10.x** - Vue 组合式工具集，核心使用 `useWebSocket` 处理实时通信断连与心跳
* **ofetch** - Nuxt 3 内置同构请求库，用于对接后端的 RESTful API
* **lodash-es ^4.17.21** - 按需加载的实用工具库（防抖、节流、深拷贝）

### 开发工具

* **ESLint ^8.x** - 统一代码检查工具
* **Prettier ^3.x** - 代码格式化工具
* **@commitlint/cli** - Git 提交信息规范检查
* **husky** - Git hooks 管理

## Project Conventions

### 代码风格

* **语言**：TypeScript 严格模式
* **引号**：单引号优先
* **命名约定**：
* 组件文件：PascalCase (如 `PdfViewer.vue`, `UploadZone.vue`)
* 组合式函数：camelCase (如 `useTaskTracker.ts`, `usePdfRender.ts`)
* 常量：UPPER_SNAKE_CASE (如 `WS_HEARTBEAT_INTERVAL`)

* **代码限制**：
* 避免在 Nuxt 服务端生命周期中调用任何涉及 `window` 或 `document` 的 DOM API
* 禁用直接操作 DOM，必须通过 Vue Refs 或专门的渲染函数实现

### 架构模式

* **组合式 API**：全面拥抱 Vue 3 Composition API `<script setup>`
* **状态管理**：按业务域划分 Pinia Store（如 `taskStore`, `profileStore`）
* **Nuxt 目录结构**：
* `/components/` - 自动导入的基础组件和业务组件
* `/pages/` - 基于文件的路由系统（如工作台主页、配置管理页）
* `/composables/` - 业务逻辑和状态提取的 Hooks
* `/plugins/` - 客户端专属插件注册（如 `pdfjs.client.ts`）
* `/assets/` - 全局样式变量（克莱因蓝/爱马仕橙主题色注入）

### 渲染与加载策略

* **SSR 与 CSR 混合**：常规页面框架利用 Nuxt 的 SSR 优势极速直出，涉及 PDF 渲染的核心组件 `<PdfViewer />` 必须包裹在 `<ClientOnly>` 中，强制采用客户端渲染以避免水合不匹配（Hydration Mismatch）。

### Git 工作流

* **提交规范**：遵循 Conventional Commits (feat, fix, docs, style, refactor 等)
* **提交检查**：通过 husky 触发 commitlint 拦截不合规的 commit message。

## Domain Context

### PDF 高亮渲染领域知识

* **Canvas vs DOM**：PDF 本体使用 Canvas 渲染以保证高保真度，交互层（选中文本、高亮框）必须使用 DOM 层叠加在 Canvas 之上。
* **坐标映射机制**：需解决后端返回的提取文本（如 "苹果", "240.6"）到 PDF 物理坐标 (x, y, width, height) 的逆向定位问题。

### 业务场景

* **智能化文档审阅**：用户拖拽上传研报，系统自动提取公司名称、数值数据并高亮显示，用户点击高亮区域可查看 LLM 提取的结构化详情。
* **配置档案切换**：针对不同类型的文档（如手机市场报告、财务报表），动态切换 YAML 配置以改变提取规则和高亮颜色。

## Important Constraints

### 技术约束

* **内存泄漏防护**：PDF.js 在频繁切换大体积文档时极易产生内存泄漏，必须在组件 `onBeforeUnmount` 阶段手动销毁 `pdfDocument` 实例并清空 Canvas 画布。
* **Worker 线程隔离**：PDF.js 的解析计算任务 (`pdf.worker.js`) 必须配置为独立 Worker 线程加载，严禁阻塞前端 UI 主线程。

### 业务约束

* **API 强依赖**：前端深度依赖后端的 MinerU 解析状态和 LangExtract 提取结果，需做好后端服务不可用时的优雅降级与错误提示。
* **视觉一致性**：必须严格遵循深色科技风的设计规范，所有动态生成的高亮 `<mark>` 标签需适配整体 UI 主题。

## External Dependencies

### 核心服务对接

* **REST API**：对接 FastAPI 服务端提供的任务提交 (`/api/v1/tasks`)、状态查询与配置管理接口。
* **WebSocket 推送**：对接 `ws://localhost:8000/ws/{task_id}` 获取任务流转的实时日志和进度反馈。

### 资源依赖

* **静态资源托管**：由前后端分离架构决定，前端静态文件可通过 Nginx 部署，或直接交由 FastAPI 的静态文件服务路由代理。
* **字体渲染**：为确保 PDF 文本层与 Canvas 渲染对齐，可能需要引入或预加载标准的中文字体包。
